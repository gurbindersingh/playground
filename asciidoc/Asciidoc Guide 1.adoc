[.lead]
This guide is a practical, reference-style overview of AsciiDoc authoring.
It covers the core language, the features most teams actually use, and the major implementation-specific features you are likely to meet in Asciidoctor.

IMPORTANT: There is no single perfectly uniform implementation of AsciiDoc.
This guide distinguishes between portable syntax and processor-specific behavior when that distinction matters.

== Scope

This guide aims to cover the mainstream authoring surface of AsciiDoc comprehensively:

* document structure
* sections and metadata
* paragraphs and inline markup
* lists
* admonitions
* delimited blocks
* links and cross-references
* attributes
* includes and conditionals
* tables
* images and media
* source code, callouts, and syntax highlighting
* footnotes, bibliography, glossary, and index terms
* STEM, passthroughs, roles, options, and backend-specific features

Where behavior is implementation-specific, examples assume Asciidoctor unless stated otherwise.

== Quick start

A minimal document:

....
= My Document
Author Name
:toc:

== Introduction

AsciiDoc turns plain text into structured output.

* It supports lists
* links: https://example.com[Example]
* and inline formatting like *bold* and _emphasis_
....

A typical command-line workflow with Asciidoctor:

[source,bash]
-----
asciidoctor mydoc.adoc
asciidoctor-pdf mydoc.adoc
asciidoctor -b docbook mydoc.adoc
-----

== Mental model

AsciiDoc is line-oriented and context-sensitive.
Most constructs are created by one of four mechanisms:

* prefixes, such as `==` for sections or `*` for list items
* block delimiters, such as `----` or `====`
* macros, such as `image::diagram.png[]`
* attributes, such as `:toc:` or `{product-name}`

A useful rule is to separate three things in your head:

. *Structure* such as sections, lists, and tables
. *Rendering controls* such as attributes, roles, and options
. *Backend-specific behavior* such as raw HTML, docinfo, or PDF page breaks

== Processor landscape

The language is broader than any one processor.
In practice, most current documentation workflows use Asciidoctor.

A portability model that works well:

* treat headings, paragraphs, lists, anchors, links, images, tables, and basic blocks as portable
* treat attributes, includes, and conditionals as common but somewhat processor-dependent
* treat UI macros, extension APIs, raw passthroughs, docinfo, and advanced publishing hooks as Asciidoctor-centric

== Document header and metadata

=== Title

The document title usually appears on the first line:

....
= Project Handbook
....

=== Author and revision lines

Classic header form:

....
= Project Handbook
Ada Lovelace <ada@example.com>
v2.3, 2026-03-22: Editorial refresh
....

How revision fields are parsed can vary slightly by processor, but this form is common.

=== Header attributes

Header attributes are declared before the first section.
They configure rendering and expose metadata.

....
= Project Handbook
:toc:
:toclevels: 3
:sectnums:
:icons: font
:source-highlighter: rouge
:description: Internal engineering handbook
:keywords: engineering, handbook, operations
....

Common attributes:

[cols="1,3"]
|===
|Attribute |Purpose

|`toc`
|Enable a table of contents.

|`toclevels`
|Set maximum depth shown in the table of contents.

|`sectnums`
|Enable automatic section numbering.

|`sectanchors`
|Add anchor links to section titles in supported backends.

|`icons`
|Select icon behavior for admonitions.

|`source-highlighter`
|Enable syntax highlighting integration.

|`doctype`
|Choose document type such as `article`, `book`, or `manpage`.

|`experimental`
|Enable selected experimental syntax in Asciidoctor, including UI macros.

|`reproducible`
|Reduce timestamp and build-environment variance.
|===

=== Setting, unsetting, and using attributes

Set an attribute:

....
:project-name: Atlas
....

Set an empty attribute:

....
:toc:
....

Unset an attribute:

....
:project-name!:
....

Reference an attribute inline:

....
Welcome to {project-name}.
....

=== Attribute precedence

In a typical Asciidoctor workflow, precedence is roughly:

. attributes passed by the CLI or API
. document-defined attributes
. processor defaults

CLI/API attributes may be locked, meaning the document cannot override them.

== Paragraphs and line handling

=== Normal paragraphs

Plain text separated by blank lines becomes paragraphs.
A soft wrap in source does not force a line break in output.

....
This is one paragraph that may
span multiple source lines.

This is a new paragraph.
....

=== Forced line breaks

A line ending with ` +` forces a break in the rendered paragraph.

....
Line one +
Line two
....

Another approach is `[%hardbreaks]`:

....
[%hardbreaks]
This block
preserves line breaks.
....

=== Literal paragraphs

Indenting a paragraph makes it literal in classic syntax:

....
 This is a literal paragraph.
 It preserves spaces and line breaks.
....

For maintainability, explicit literal or listing blocks are usually clearer.

A lone `+` continues the current list item with another block.
This is essential when attaching paragraphs, source blocks, tables, or admonitions to a list item.

== IDs, roles, and options

Blocks can carry an attribute list.
Common forms:

....
[#id]
[.role]
[%option]
[quote, Ada Lovelace, Notes]
....

Combined form:

....
[#intro.lead%collapsible]
== Introduction
....

Practical meaning:

* IDs create stable anchors
* roles add semantic or styling hooks
* options toggle processor- or backend-specific behavior

== Delimited blocks

Delimited blocks are one of the core mechanisms in AsciiDoc.

[cols="1,1,3"]
|===
|Delimiter |Name |Purpose

|`----`
|listing or source
|Verbatim text, code, or terminal output

|`....`
|literal
|Verbatim text without source semantics

|`====`
|example
|Generic boxed example content

|`****`
|sidebar
|Sidebar or aside content

|`____`
|quote
|Long quotations or verse-style content

|`++++`
|pass
|Raw backend content or passthrough

|`--`
|open
|Generic container block

|`////`
|comment
|Hidden content
|===

=== Block titles

A block title is a line beginning with `.` directly above the block.

.....
.Sample output
----
Hello, world.
----
.....

=== Example block

.....
.An example block
====
This content is grouped as an example.
====
.....

=== Sidebar block

.....
.Optional background
****
A sidebar is visually separate from the main flow.
****
.....

=== Quote block

.....
[quote, Grace Hopper, Interview]
____
The most dangerous phrase in the language is,
"We've always done it this way."
____
.....

=== Verse block

.....
[verse, Author Name, Work Title]
____
First line
Second line
Third line
____
.....

=== Listing block

.....
----
verbatim text
with spaces preserved
----
.....

=== Source block

.....
[source,python]
----
def greet(name: str) -> None:
    print(f"Hello, {name}")
----
.....

Useful options:

.....
[source,java,linenums]
----
class Main {}
----

[source,python,subs=+attributes]
----
print("{project-name}")
----
.....

=== Literal block

.....
....
Literal content
is preserved exactly.
....
.....

=== Open block

.....
[abstract]
--
This open block is styled as an abstract.
--
.....

=== Passthrough block

.....

== Substitutions

AsciiDoc transforms content through substitution groups such as:

* special characters
* quote replacements
* attributes
* macros
* post replacements

The active substitutions depend on context.
Normal paragraphs expand attributes and macros.
Literal-style contexts typically do not.
Passthrough contexts suppress most or all normal substitutions.

=== Custom substitutions on a block

.....
[subs="attributes,macros"]
++++
Content with a custom substitution pipeline.
++++
.....

A common source-block pattern:

.....
[source,xml,subs=+attributes]
----
<service name="{project-name}"/>
----
.....

Use custom substitution pipelines only when necessary.
They are powerful but easy to misuse.

== Includes

=== Basic include

....
include::partial/intro.adoc[]
....

=== Include with level offset

....
include::chapters/install.adoc[leveloffset=+1]
....

=== Tagged includes

File with tags:

....
// tag::summary[]
This is the summary.
// end::summary[]
....

Parent file:

....
include::partials/overview.adoc[tag=summary]
....

=== Include line ranges

....

....

=== Include safety

Include behavior is often restricted by safe mode, base directory, or hosted-renderer policy.
If an include works locally but not in CI or a portal, safe-mode restrictions are a likely cause.

== Conditionals

Conditionals are widely used in Asciidoctor to target multiple outputs from one source.

=== ifdef and ifndef

.....
ifdef::backend-html5[]
This appears only in HTML5 output.
endif::[]

ifndef::env-github[]
This appears outside GitHub rendering.
endif::[]
.....

=== ifeval

.....
:edition: enterprise

ifeval::["{edition}" == "enterprise"]
Enterprise-only content.
endif::[]
.....

Conditionals increase maintenance cost.
Use them where source reuse is worth the complexity.

|Name |Role

....

|===

....

|Column |Meaning

....

....

....
|===

* another item

....

The `a|` style is the key to nested AsciiDoc content inside a table cell.

== Safe modes and security

Safe modes may restrict:

* include targets
* URI reading
* docinfo injection
* raw content handling

This matters in CI, hosted renderers, and documentation portals.
If a construct works locally but fails in production, safe mode or sandboxing is a likely explanation.

== Backend-specific behavior

The same source can render differently in HTML, PDF, EPUB, DocBook, or manpage output.
Common differences include:

* icon rendering
* syntax highlighting support
* page breaks
* multimedia behavior
* CSS or theme handling
* index and bibliography generation
* raw passthrough handling

Always test the exact backend and theme stack that will publish the document.

== GitHub and lightweight renderers

Not all renderers implement full AsciiDoc.
Hosted platforms may disable or ignore:

* includes
* custom extensions
* advanced roles
* raw HTML or scripts
* PDF-oriented semantics

For repository READMEs, a conservative subset is usually the safest choice.

== Docinfo and template customization

Asciidoctor can inject custom header or footer fragments through docinfo files in some workflows.
Typical uses:

* custom CSS or JavaScript
* analytics snippets
* metadata injection
* site-specific wrappers

This is build-layer customization, not core content syntax.

== Extensions and plugins

Modern Asciidoctor supports extensions that can add:

* custom block macros
* include processors
* tree processors
* postprocessors
* custom syntax handling

Extensions are powerful, but they sharply reduce portability.
Treat them as platform features, not language basics.

== Content architecture for large docs

For larger documentation systems:

* store reusable fragments in `partials/`
* use `include::[]` to assemble deliverables
* expose configuration through attributes
* isolate backend differences behind conditionals
* define explicit anchors for stable xrefs

A modular pattern:

....
= Platform Guide
:toc:

include::partials/intro.adoc[]
include::chapters/install.adoc[leveloffset=+1]
include::chapters/operate.adoc[leveloffset=+1]
....

== Troubleshooting

=== Formatting did not apply

Check:

. whether the markers are constrained by adjacent word characters
. whether the text is inside a literal, listing, or passthrough context
. whether a custom substitution pipeline disabled the relevant substitution group

=== Lists are broken

Most list failures come from:

* missing blank lines
* missing continuation `+`
* blocks attached to the wrong list item
* accidental indentation creating a literal paragraph

=== Includes fail

Likely causes:

* wrong relative path
* safe-mode restrictions
* base-directory mismatch
* hosted renderer with include disabled

=== Xrefs do not resolve

Check:

* the target ID exists
* the implicit ID is what you think it is
* the inter-document path is correct
* the current site generator rewrites xrefs differently

=== Tables look malformed

Check:

* balanced delimiter lines
* correct `cols` declaration
* span values that do not exceed table width
* use of `a|` where nested AsciiDoc is required

== Authoring guidance

=== Prefer explicit syntax

Good:

.....
[source,yaml]
----
name: app
----
.....

More fragile:

....
 name: app
....

Explicit blocks are easier to maintain than indentation-driven shortcuts.

=== Use explicit IDs for stable links

Implicit IDs change when headings change.
If links matter, define the IDs yourself.

=== Minimize raw backend markup

Use raw HTML or passthrough only when no higher-level construct works.
That preserves portability.

=== Centralize configuration in attributes

....
:product-name: Atlas
:product-version: 3.2
....

Then reference them consistently:

....
Welcome to {product-name} {product-version}.
....

=== Keep include graphs understandable

Modularity is good.
Deeply nested include hierarchies are difficult to debug.

== Practical patterns

=== Reusable warning text

....
:danger-note: Do not run this command in production.

WARNING: {danger-note}
....

=== Product-edition filtering

.....
:edition: community

ifeval::["{edition}" == "enterprise"]
This feature is available only in enterprise editions.
endif::[]
.....

=== Shared code snippets with tagged includes

.....
[source,python]
----

----
.....

=== Admonition inside a list item

.....

+
[WARNING]
====
Do not stop the service during a schema migration.
====
. Apply the update.
.....

== Final guidance

Use the smallest feature set that accomplishes the publishing goal.
AsciiDoc scales from simple README files to book-length documentation systems, but maintainability depends on disciplined use of attributes, includes, anchors, and backend-specific features.

When portability matters:

* prefer plain structural syntax
* avoid raw backend markup
* define explicit anchors
* keep conditionals limited and documented
* test every target backend in CI

When power matters more than portability:

* leverage Asciidoctor attributes and extensions
* use tagged includes for source reuse
* theme by backend rather than hardcoding presentation in content
* treat the document source as part of the build system, not just prose
